- DONE — Added `GET /api/1/laps/aggregate?groupBy={track|day|week|month}` server-side `COUNT(*) GROUP BY` endpoint. `OverviewScreen` now drives **Laps per week/month** and **Tracks practiced** via this aggregate; the 1000-row `lapsQuery` is gone.
- DONE — Added `GET /api/1/sessions/aggregate?groupBy={day|week|month}` that returns per-bucket `count` and `drivingTimeMs`. `OverviewScreen` now drives the **Sessions per …**, **Driving time per …** and **Driving Time** stat card from this single request instead of reducing `sessionsQuery.items` client-side.
- DONE — The streak now derives from the all-time `GET /api/1/sessions/aggregate?groupBy=day` response (one count-only row per active day) via `streakFromDayKeys`, instead of iterating up to 100 raw session items. - `frontend/` has **two identical Vite configs** — `vite.config.js` and `vite.config.ts` — both tracked in git. Vite resolves `.js` before `.ts`, so the `.js` is the one actually loaded and the `.ts` is dead. The `__APP_VERSION__` `define` (added for the feedback form) had to be duplicated into both to keep them in sync. Delete `vite.config.js` and keep only the typed config (confirm nothing references the `.js` path).
- Feedback form is **build-time gated** (`VITE_FEEDBACK_*` are inlined by `vite build`, not read at runtime). Real values live in the committed `frontend/.env.production`, but they're currently blank, and there is **no distribution build** that actually ships the frontend yet — CI (`build-and-test.yml`) only runs `pnpm build` as a check and discards the output, and the backend doesn't serve the built assets. Action: fill in `frontend/.env.production` from the maintainer's Google Form, and wire a real release/distribution build (or have the backend serve `frontend/dist`) so downloaders get a bundle with the launcher enabled.
- Feedback form (`frontend/src/components/feedback/FeedbackButton.tsx`) submits via an unauthenticated `no-cors` Google Form POST with **no spam protection** and **no delivery confirmation** (an opaque response is assumed successful) — same limitations as the landing-page register-interest form. If abused, add a honeypot / Turnstile / Worker proxy. Wrong or edited Google Form `entry.<id>`s silently drop fields since the POST can't validate them.

- Lap telemetry linkage was **never back-filled** for laps recorded before this fix. `ClientInitializer` recorded every REALTIME_CAR_UPDATE frame with a null `lapUid`, and nothing stamped the owning lap on completion — so `findByLapUid` returned nothing and every lap comparison / telemetry trace was empty. Fixed going forward: on `LAPCOMPLETED`, `LinkLapTelemetryService.linkCompletedLap` stamps the new lap's id/uid onto that car's still-unlinked frames (`sessionUid + carIndex + laps < lapNumber`). **Remaining debt:** existing rows in already-captured databases still have null `lap_uid` and won't show telemetry — needs a one-time migration that links historical frames to laps by `(session_id, car_index, laps → lap_number)`. Until then, only laps recorded after the fix render telemetry.
- Missing-car on laps is now corrected going forward: a lap that completes before its `EntryListCar` is processed is recorded with a null `car`, and `RecordCarOnLapsService.fillMissingCar` now back-fills it when the model is registered (`UPDATE LAP SET car WHERE session_uid + car_id AND car IS NULL`), plus the `EntryListCar` registration is no longer gated on the focused car being known. **Remaining debt:** laps already stored with a null `car` in existing databases aren't repaired — needs a one-time migration (e.g. re-derive from the car's other laps in the same session, or from retained `EntryListCar` data) to fill historical null `car` values.
